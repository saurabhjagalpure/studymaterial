Introduction to RAG

Retrieval-Augmented Generation combines the power of LLMs with external knowledge — the foundation of every modern AI app.

🧠

What is RAG?

The core idea

RAG stands for Retrieval-Augmented Generation. The idea is simple: instead of relying only on what an LLM memorized during training, we first retrieve relevant documents from an external knowledge base, then augment the prompt with those documents, and finally let the LLM generate an answer grounded in that context.

Simple analogy

Imagine an exam with open books. Without RAG, the student answers from memory alone. With RAG, the student first finds the relevant chapter, reads it, then writes the answer. The answer is more accurate and grounded.

1

User query arrives — e.g., "What is our refund policy?"

2

Retrieve — search the knowledge base for relevant documents.

3

Augment — add retrieved docs to the prompt as context.

4

Generate — LLM produces a grounded, accurate answer.

Core concept Always asked
❓

Why RAG over Fine-tuning?

The key trade-off

Interviewers love this question. Both RAG and fine-tuning adapt LLMs to specific domains, but they solve different problems.

AspectRAGFine-tuning
Knowledge updateReal-time — just update the DBNeed to retrain the model
CostLow — no GPU training neededHigh — requires training runs
HallucinationLower — grounded in docsStill possible
Best forFactual Q&A, dynamic dataStyle/tone/format adaptation
💡

In interviews, say: "I'd use RAG when the knowledge needs to stay current or when we can't afford retraining. I'd fine-tune when I need the model to behave differently — e.g., adopt a persona or a specific output format."

⚠️

Failure Modes of Naive RAG

What goes wrong and why

Basic RAG (embed → retrieve → generate) has several well-known failure points. Understanding these is what separates mid-level from senior candidates.

1

Poor retrieval — the wrong documents are fetched; the LLM answers from irrelevant context.

2

Chunking mismatch — chunks are too large (noise) or too small (missing context).

3

Lost in the middle — LLMs tend to ignore middle tokens in long contexts.

4

Semantic mismatch — user asks a question but docs contain declarative statements; cosine similarity fails.

5

No grounding verification — LLM ignores retrieved context and still hallucinates.

Interview Q&A Questions on RAG Basics
Q1. Explain RAG in one minute to a non-technical interviewer.
+

Imagine you're a doctor who needs to answer a patient's question about a rare condition. You could either answer from memory (risky) or quickly look up the latest medical guidelines and then answer. RAG is the second approach — the LLM "looks up" relevant information from a database before answering. This makes answers more accurate, up-to-date, and verifiable.

Q2. What are the three main components of a RAG pipeline?
+

1. Indexing — documents are chunked, embedded into vectors, and stored in a vector database (e.g., Pinecone, ChromaDB, Weaviate).

2. Retrieval — at query time, the query is embedded and the most similar chunks are retrieved using approximate nearest-neighbor (ANN) search.

3. Generation — the retrieved chunks are injected into the LLM prompt as context, and the LLM generates the final answer.

Q3. When would you NOT use RAG? What are its limitations?
+

RAG is not ideal when: (a) the task requires deep reasoning that spans many documents and can't be captured in top-k chunks; (b) latency is critical — retrieval adds network + compute overhead; (c) your knowledge is fully static and small enough to fine-tune or fit in a system prompt; (d) you need the model to internalize a completely new format or behavior, not just factual knowledge.

Q4. What is the difference between RAG and a search engine?
+

A search engine retrieves and ranks documents, returning links. RAG goes further — it retrieves documents and then synthesizes a natural language answer from them using an LLM. Search gives you "where to look"; RAG gives you the answer. Also, search uses keyword/BM25 matching while RAG typically uses semantic (vector) search.

Query Translation

User queries are often ambiguous or poorly phrased. Query translation transforms them into better retrieval signals.

🔄

Multi-Query Approach

Generate multiple retrieval queries from one user question

A single user query often misses relevant documents because it's phrased in a specific way. Multi-query generates several reformulations of the same question and retrieves documents for each, then deduplicates results.

Example

User: "How do I fix slow SQL queries?"

Generated queries:
→ "SQL query optimization techniques"
→ "Database performance tuning best practices"
→ "How to use query execution plans"
→ "Indexing strategies for faster database queries"

Each query retrieves different chunks. The union gives broader, more complete coverage. Great for complex questions.

⚡

RAG Fusion

Combine multiple retrievals using Reciprocal Rank Fusion

RAG Fusion extends multi-query by using Reciprocal Rank Fusion (RRF) to merge and re-rank results from multiple retrieval runs into a single ranked list.

RRF formula

score(doc) = Σ 1 / (rank_in_list_i + k)

A document ranked #1 in list A and #3 in list B scores higher than one ranked #5 in both. k is typically 60 (dampens extreme ranks).

1

Generate N sub-queries from the original.

2

Retrieve top-k docs for each sub-query.

3

Apply RRF to all retrieval lists → one ranked list.

4

Feed top results to the LLM.

🌿

Decomposition

Break complex questions into sub-questions

For multi-hop questions (questions that require connecting information from multiple places), decomposition breaks one big question into smaller, answerable sub-questions.

Example

Original: "What are the tax implications of selling RSUs if I'm based in California?"

Decomposed into:
→ "How are RSUs taxed federally in the US?"
→ "What is California state income tax on stock compensation?"
→ "What is the difference between RSU vesting and sale tax events?"

Each sub-question retrieves its own context. Answers are assembled bottom-up (sequential or parallel).

🦢

HyDE — Hypothetical Document Embeddings

Generate a fake answer, embed it, retrieve with that

HyDE solves the query-document semantic mismatch problem. Queries are short questions; documents contain declarative answers. Their embeddings differ in vector space.

Solution: Ask the LLM to hallucinate a plausible answer to the query. Embed that hypothetical answer, and use it to search the database — it lands closer to real documents than the original question embedding would.

Example

Query: "What is the boiling point of ethanol?"

HyDE generates: "Ethanol (C2H5OH) has a boiling point of 78.37°C at standard atmospheric pressure…"

This hypothetical doc embeds near actual chemistry docs even though the LLM made it up. Retrieval quality improves significantly.

Clever technique Frequently asked
🔗

IR-CoT — Interleaved Retrieval with Chain of Thought

Retrieve → think → retrieve again

IR-CoT interleaves retrieval steps with reasoning steps. After each reasoning step, if the model realizes it needs more information, it triggers another retrieval. This is iterative and very powerful for complex multi-hop questions.

1

Retrieve initial context.

2

LLM reasons one step (CoT).

3

If more info is needed, new retrieval triggered.

4

Repeat until answer is reached.

vs Adaptive RAG

IR-CoT always interleaves. Adaptive RAG decides whether to retrieve at all. IR-CoT is about the how; Adaptive RAG is about the when.

Interview Q&A Query Translation Questions
Q1. What problem does HyDE solve and when would you use it?
+

HyDE solves the semantic gap between queries and documents. Queries are short and question-like; documents are long and declarative. Their vector representations are in different parts of embedding space, so cosine similarity between them is low even when they're semantically related.

HyDE generates a hypothetical answer and embeds that instead of the query. This works because the hypothetical answer is in the same "style" as real documents, so it retrieves more relevant chunks. Use it when your queries are very short/telegraphic and you notice retrieval quality is poor.

Q2. How does RAG Fusion differ from simply running multi-query?
+

Multi-query retrieves documents for each sub-query and takes the union — but this loses ranking information. A document that appears #1 in three different query results is treated the same as one that appears once at #10.

RAG Fusion adds Reciprocal Rank Fusion (RRF) to merge the ranked lists intelligently. Documents that consistently rank high across multiple retrieval runs get a boosted score, giving a much better final ranking without needing expensive cross-encoder re-ranking.

Q3. What is the risk of using HyDE and how do you mitigate it?
+

The main risk is hallucination in the hypothetical document. If the LLM hallucinates a convincing but incorrect hypothetical answer, the embedding will point to wrong documents. For niche domains where the LLM has little training data, HyDE can actually hurt retrieval.

Mitigation: (1) Use HyDE only when your LLM has reasonable knowledge of the domain. (2) Run HyDE alongside standard query embedding and combine both retrieval results (ensemble approach). (3) Evaluate HyDE vs. baseline on your specific dataset before deploying.

Q4. Explain query decomposition with a real-world example.
+

Say a user asks: "Compare the revenue growth of Apple and Google in 2023 and explain what drove the difference."

This is a multi-hop question. Decomposed: (1) "Apple revenue growth 2023" → retrieve Apple financials. (2) "Google revenue growth 2023" → retrieve Alphabet reports. (3) "Key drivers of tech revenue growth 2023" → retrieve analysis. Then synthesize all three answers. Without decomposition, a single query might retrieve generic tech articles that don't answer either specific question well.

Chunking Strategies

Chunking is how you split documents before indexing. It's one of the most impactful and most overlooked decisions in a RAG pipeline.

📏

Fixed-Size Chunking

Split by character/token count

The simplest strategy: split text every N characters or tokens, optionally with an overlap between consecutive chunks to preserve context at boundaries.

Example (chunk_size=100, overlap=20)

Document: "The cat sat on the mat. The dog ran away. The bird flew high."

Chunk 1: "The cat sat on the mat. The dog ran away."
Chunk 2: "The dog ran away. The bird flew high." ← overlap keeps context

Pros: Simple, fast, predictable size. Cons: Blindly cuts mid-sentence, mid-thought, or mid-table. Overlap helps but doesn't eliminate the problem.

SimpleContext bleed
🔀

Recursive Character Splitting

Split on natural boundaries first

Used by LangChain's RecursiveCharacterTextSplitter. It tries a hierarchy of separators in order: paragraphs (\n\n) → sentences (\n) → words ( ) → characters. Only moves to the finer level if chunks are still too large.

Priority order

1st try split on: \n\n (paragraphs)
If still too big, split on: \n (lines)
If still too big, split on: . (sentences)
Last resort: split on: " " (words)

This preserves semantic units much better than fixed-size chunking. Almost always a better default.

🧬

Semantic Chunking

Split where meaning changes

Embed every sentence. Then find points where the embedding distance between consecutive sentences spikes — these are topic boundaries. Chunk at those boundaries. Each chunk is semantically coherent.

1

Split document into individual sentences.

2

Embed each sentence with an embedding model.

3

Compute cosine distance between consecutive sentence embeddings.

4

Where distance > threshold, create a new chunk boundary.

Pros: Chunks are topically coherent. Cons: More expensive — requires embedding every sentence during indexing.

Best qualitySlower
📐

Chunk Size Trade-offs

The most important decision you'll make

Chunk SizeProsCons
Small (128–256 tokens)Precise retrieval, less noiseMissing context, incomplete answers
Medium (512–1024 tokens)Good balanceMore noise than small
Large (2048+ tokens)Full context preservedNoisy, dilutes relevance signal
💡

In interviews, say: "Chunk size is a hyperparameter I'd tune empirically. I'd start at 512 tokens with 10–20% overlap, measure retrieval precision/recall, then adjust."

Interview Q&A Chunking Questions
Q1. How do you choose chunk size for a RAG pipeline?
+

It depends on three factors: (1) Document type — legal docs need larger chunks (full clauses); FAQs work well with smaller ones. (2) LLM context window — you need to fit top-k chunks + prompt + answer in the window. (3) Query type — factual lookups need small precise chunks; analytical questions need larger context.

In practice, I'd start with 512–768 tokens and 15% overlap, evaluate retrieval quality using metrics like MRR and Context Recall, then tune.

Q2. What is the "lost in the middle" problem and how does chunking affect it?
+

Research shows LLMs pay more attention to information at the beginning and end of the context window, and less to the middle. If you retrieve 10 chunks and inject them all, the chunks in positions 3–7 are often ignored.

Chunking affects this because smaller chunks mean more chunks to fill the same window — increasing the chance of relevant info landing in the middle. Solutions: use fewer but better chunks (better retrieval), re-rank so the best chunk is first, or use "long context" models less susceptible to this.

Q3. How do you handle chunking for structured data like tables or code?
+

Standard text splitting destroys structure. For tables: keep each row or logical section as a single chunk and prepend the column headers; alternatively convert tables to natural language summaries. For code: use AST-aware splitting (split on function/class boundaries, not lines). For PDFs: use layout-aware parsers (PyMuPDF, Unstructured.io) that recognize headers, columns, and tables before chunking.

Indexing Strategies

Indexing determines how documents are stored and retrieved. The right indexing strategy can dramatically improve retrieval quality.

📦

Single Vector Embedding

One vector per chunk

The baseline approach: embed each chunk into a single dense vector (e.g., 1536 dimensions with OpenAI's text-embedding-3-small). Store in a vector DB. At query time, embed the query and find the nearest vectors via cosine similarity or dot product.

How it works

Chunk → embedding model → [0.12, -0.45, 0.87, …] (1536 floats)
Query → embedding model → [0.11, -0.43, 0.89, …]
similarity = cosine(query_vec, chunk_vec) → 0.92 ✓

Limitation: One vector must capture the entire meaning of a chunk. Long chunks lose nuance. A question about a detail in a 1000-token chunk may not retrieve well.

🗂️

Multi-Vector Embedding

Multiple representations per document

Instead of one vector per chunk, generate multiple vectors for each document: the full chunk, a summary, individual key sentences, or hypothetical questions the chunk answers. Store all vectors but link them back to the original chunk. Retrieval finds any vector; you return the full chunk.

1

For each chunk, generate: a summary embedding, a full-text embedding, and question embeddings.

2

Store all embeddings in the vector DB, all pointing to the same chunk ID.

3

A query matches any of these embeddings → retrieves the full chunk.

Better recallMore storage
🌲

RAPTOR — Recursive Abstractive Processing for Tree-Organized Retrieval

Build a summary tree over your corpus

RAPTOR builds a hierarchical tree of summaries. Leaf nodes are raw chunks. Parent nodes are LLM-generated summaries of clusters of leaves. The root is a summary of everything. Retrieval happens at all levels simultaneously.

1

Embed all chunks. Cluster similar chunks (e.g., k-means on embeddings).

2

LLM summarizes each cluster → new "parent" nodes.

3

Repeat clustering+summarizing on parent nodes until root.

4

At query time, search across all levels — get both high-level and fine-grained results.

Why RAPTOR helps

Questions like "give me a company overview" benefit from high-level summaries. Questions like "what is the Q3 revenue?" need leaf-level detail. RAPTOR handles both.

🎯

ColBERT — Contextualized Late Interaction over BERT

Token-level matching for precise retrieval

Standard dense retrieval compresses a document into one vector — losing fine-grained token information. ColBERT keeps a vector for every token in the document, and scores query-document similarity using the MaxSim operator: for each query token, find its most similar document token, then sum those max scores.

MaxSim scoring

Query tokens: ["what", "is", "dropout"]
For "dropout" → find most similar doc token → "regularization" (score 0.91)
For "what" → "definition" (score 0.82)
Total score = sum of max similarities

ColBERT achieves near cross-encoder accuracy with near bi-encoder speed. Downside: storage is much larger (one vector per token per doc).

High accuracyMore storage
🔀

Hybrid Search — Dense + Sparse (BM25)

Combine semantic and keyword search

Dense retrieval excels at semantic matching but misses exact keyword matches. BM25 (a TF-IDF variant) excels at keyword matching but fails for paraphrases. Hybrid search combines both.

Dense (Embeddings)Sparse (BM25)
StrengthSemantic similarityExact keyword match
Example"car" matches "automobile""GPT-4" matches "GPT-4" exactly
Fails onRare terms, product codesParaphrases, synonyms

Hybrid scoring: score = α × dense_score + (1-α) × bm25_score. Tune α on your dataset. Most production RAG systems use hybrid search.

Interview Q&A Indexing Questions
Q1. What is the difference between bi-encoder and cross-encoder retrieval?
+

A bi-encoder embeds the query and document independently into vectors, then computes similarity with a dot product. This is fast because document vectors are precomputed and stored — you only embed the query at runtime. But it's less accurate because query and document never "see" each other during encoding.

A cross-encoder takes the query+document concatenated as a single input, letting the model attend across both simultaneously. This is much more accurate but slow — you can't precompute scores. In practice: use bi-encoders (dense retrieval) for candidate retrieval, then cross-encoders for re-ranking the top candidates.

Q2. Why is hybrid search better than pure dense retrieval?
+

Pure dense retrieval fails on rare terms and exact matches. If a user queries "CVE-2024-12345" or "SK-U7729-B", there's no meaningful semantic neighborhood for these. The embedding model was never trained on them. BM25 handles these perfectly with exact matching.

Hybrid also handles domain-specific jargon better. A medical query for "BRCA1 mutation" might embed near general biology terms, missing the exact BRCA1 papers. BM25 finds exact matches immediately. Studies show hybrid consistently outperforms either alone by 5–15% on standard benchmarks.

Q3. Explain RAPTOR and when you'd use it over standard flat indexing.
+

Use RAPTOR when your corpus has documents covering different levels of abstraction, and your users ask both high-level ("give me an overview of X") and detail-level ("what was the Q3 margin?") questions. Flat indexing with raw chunks can't answer the high-level question well — no single chunk has the overview. RAPTOR's parent nodes (summaries) handle that.

The cost is extra LLM calls and storage during indexing. Worth it for long documents, books, annual reports, or large wikis.

Routing & Query Construction

Routing decides where to send a query. Query construction transforms natural language into the right retrieval format for the target data source.

🚦

Logical Routing

Rule-based or LLM-classified routing

Logical routing uses an LLM (or a classifier) to inspect the query and decide which data source or knowledge base to query. Think of it as an intelligent dispatcher.

Example

Query: "What is our HR leave policy?" → route to HR docs DB
Query: "How do I reset my password?" → route to IT support KB
Query: "SELECT * FROM orders WHERE..." → route to SQL tool

The router uses a small, fast LLM to classify first, then sends the query to the right retriever.

SimpleMulti-source
🧭

Semantic Routing

Embed the query, find the best data source by similarity

Each data source is described by a set of "representative" texts or example queries. These are embedded and stored. At runtime, embed the user query and find the data source whose example embeddings are closest.

Example

DB 1 description: "questions about financial statements, revenue, profit, balance sheet"
DB 2 description: "questions about product features, tutorials, how-to guides"

Query: "What was the net margin in Q4?" → embeds closer to DB 1's examples → route to finance DB

No LLM needed for routing — just embedding similarity. Faster and cheaper than logical routing, but less flexible.

🏗️

Query Construction — Self-Query / Metadata Filtering

Convert natural language into structured filters

Vector search retrieves by semantic similarity, but sometimes the user specifies exact constraints: date ranges, authors, document types, etc. Self-querying extracts these constraints from the query and converts them into structured metadata filters + a refined semantic query.

Example

User query: "Show me papers about transformers published after 2022 by researchers at Google"

Extracted:
semantic_query = "transformer architecture papers"
filters = { year: { "$gt": 2022 }, affiliation: "Google" }

Vector DB is queried with BOTH the semantic query AND the filters applied simultaneously.

Structured + semanticHigh precision
🗄️

Text-to-SQL

Turn natural language into database queries

When the data source is a relational database (not a vector DB), query construction means generating a SQL query from natural language. The LLM receives the schema + the question, and outputs valid SQL.

Example

Schema: orders(id, customer_id, amount, date, status)
Query: "How many orders were placed last month worth over $500?"

Generated SQL:
SELECT COUNT(*) FROM orders
WHERE date >= DATE_TRUNC('month', NOW() - INTERVAL '1 month')
AND amount > 500

This is commonly used in agentic RAG systems where the agent decides whether to query a vector DB, a SQL DB, or an API.

Interview Q&A Routing & Query Construction
Q1. How would you design a RAG system that handles both SQL databases and document stores?
+

Use a router + query constructor pattern. First, classify the query with a router: "Is this a structured data question (SQL) or an unstructured document question (vector search)?" For SQL questions, use an LLM with the database schema to generate SQL. For document questions, run standard vector search. For hybrid questions, run both and combine results before generation.

In practice, I'd implement this as an agent with tools: a sql_search tool, a vector_search tool, and let the LLM decide which to call. This is the basis of modern agentic RAG.

Q2. What is the difference between logical and semantic routing? When do you use each?
+

Logical routing uses an LLM to classify the intent and pick a route. It's more flexible — you can write complex routing rules in the prompt, handle edge cases, and explain decisions. Use it when the routing logic is complex or when data sources are very different.

Semantic routing uses embedding similarity — no LLM call needed for routing. Use it when you have well-defined, distinct data sources and latency matters. It's faster and cheaper but can't handle ambiguous cases as gracefully.

Post-Retrieval

You've retrieved documents — but are they the right ones, in the right order? Post-retrieval is about refining what goes into the LLM prompt.

📊

Re-ranking with Cross-Encoders

Score query+document pairs for precision

After retrieving top-k candidates via fast bi-encoder search, a cross-encoder re-ranks them. It takes (query, document) pairs and outputs a relevance score. Much more accurate than cosine similarity but too slow for initial retrieval.

1

Dense retrieval fetches top-50 candidates (fast).

2

Cross-encoder scores all 50 (query, doc) pairs.

3

Re-rank by cross-encoder score. Take top-5.

4

Feed these 5 highly-relevant chunks to LLM.

Popular cross-encoders: ms-marco-MiniLM-L-12-v2, Cohere Rerank API, Jina Reranker.

High precisionAdded latency
✂️

Contextual Compression

Extract only the relevant part of each chunk

A retrieved chunk might be 500 tokens, but only 2 sentences are actually relevant to the query. Contextual compression passes each chunk through an LLM extractor that extracts only the relevant sentences, reducing noise and token cost.

Example

Retrieved chunk: "The company was founded in 2010. It operates in 40 countries. The return policy allows 30 days for all products. Our CEO is John Smith. Revenue in 2023 was $4B."

Query: "What is the return policy?"
After compression: "The return policy allows 30 days for all products."

Dramatically reduces the context window usage. Especially useful when chunks are large or loosely structured.

🔎

LLM-based Relevance Filtering

Ask the LLM: "Is this relevant?"

Before injecting retrieved chunks into the final prompt, pass each one through a fast LLM filter with the question: "Is the following document relevant to the query: [query]? Answer yes or no." Discard chunks that say no.

When to use

Use when you notice the LLM is being distracted by irrelevant context ("garbage in, garbage out"). This adds latency but can significantly improve faithfulness scores. A small, fast model (like GPT-4o-mini) works fine for filtering.

Reduces noiseExtra LLM call
Interview Q&A Post-Retrieval Questions
Q1. When would you use a re-ranker vs. just increasing k in retrieval?
+

Increasing k gets you more candidates but doesn't improve their order or relevance quality. If the top-1 result is wrong, adding more candidates doesn't fix that — it just adds noise. A re-ranker fundamentally reorders results based on a more powerful relevance signal.

Use a re-ranker when: (a) retrieval quality is the bottleneck (correct docs are in top-50 but not top-5); (b) you can afford ~100–200ms extra latency; (c) the queries are complex or ambiguous. If latency is tight, try increasing k first, and only add a re-ranker if metrics still aren't good enough.

Q2. What is "context stuffing" and why is it a bad idea?
+

Context stuffing is retrieving a large k (e.g., top-20) and injecting all chunks into the prompt, thinking "more context = better answer." This has several downsides: (1) Token cost increases linearly with k. (2) "Lost in the middle" — LLMs ignore context in the middle of a long prompt. (3) Irrelevant chunks actively confuse the LLM (dilution effect — shown in studies to reduce accuracy). (4) Exceeds context window limits.

Best practice: retrieve top-20 as candidates, re-rank, compress, and inject only the top 3–5 highest-quality chunks.

Advanced RAG Variants

Beyond basic RAG: adaptive, corrective, self-reflective, and modular approaches that handle real-world complexity.

🎛️

Adaptive RAG

Decide whether to retrieve at all

Not every query needs retrieval. Adaptive RAG classifies queries before processing and routes them to: (a) direct LLM answer (simple factual or conversational), (b) single-step RAG, or (c) multi-step RAG (for complex queries). This saves cost and latency for simple questions.

Decision tree

"What's 2+2?" → no retrieval needed → LLM answers directly
"What is our refund policy?" → single-step RAG → retrieve → answer
"Compare our pricing vs competitors and suggest strategy" → multi-step RAG → decompose → iterative retrieval

🔁

Self-RAG

The LLM critiques its own retrieval and generation

Self-RAG trains the LLM to generate special reflection tokens that evaluate its own outputs. These tokens decide: should I retrieve? Is this retrieved doc relevant? Is my answer supported by context? Is my answer complete?

1

[Retrieve] token — LLM decides if retrieval is needed.

2

[ISREL] token — Is retrieved doc relevant? (yes/no)

3

[ISSUP] token — Is my answer supported by the doc? (yes/no)

4

[ISUSE] token — Is my answer useful? (score 1–5)

Key insight

Self-RAG doesn't use a separate grader model — the same LLM both generates and evaluates. This requires fine-tuning on special reflection token data.

✅

Corrective RAG (CRAG)

Detect bad retrieval and fall back to web search

CRAG adds a retrieval evaluator that scores how relevant the retrieved documents are. If the score is low (retrieval was poor), it falls back to web search to get better documents. If retrieval is good, it proceeds normally.

1

Retrieve from vector DB as usual.

2

Evaluator scores relevance: CORRECT / INCORRECT / AMBIGUOUS.

3

CORRECT → proceed. INCORRECT → fall back to web search. AMBIGUOUS → combine both.

4

Decompose web-retrieved results → filter → refine → generate.

RobustFallback
🧩

Modular RAG

Build RAG as composable modules

Modular RAG treats each component (retriever, router, re-ranker, generator, memory) as an independent, swappable module. This allows flexible pipeline configurations: sequential, branching, looping, or conditional.

Module examples

Search module: web, vector DB, SQL, APIs
Memory module: short-term (conversation), long-term (user profile)
Fusion module: merge results from multiple retrievers
Predict module: generates a sub-query or hypothesis
Task adaptor: adapts output format per use case

This is the framework behind LangGraph, LlamaIndex workflows, and most production agentic RAG systems.

🤖

Agentic RAG

The LLM drives the entire retrieval strategy

In Agentic RAG, the LLM acts as an agent with retrieval tools. It decides what to retrieve, when, how many times, and from where. This enables dynamic, multi-hop, and self-correcting retrieval strategies without hard-coded pipelines.

1

Agent receives the query and a set of tools (vector search, SQL, web, calculator).

2

Agent plans: "I need to first look up X, then compute Y."

3

Agent calls tools iteratively, reading results after each call.

4

Once enough context is gathered, agent generates the final answer.

Interview Q&A RAG Variants Questions
Q1. Compare Self-RAG and Corrective RAG. When would you use each?
+

Self-RAG is a fine-tuned model that adds self-reflection tokens during generation. It requires retraining the LLM and is deeply integrated into the generation process. Best when you own the model and can fine-tune it, and you need inference-time control at the token level.

CRAG is a pipeline-level technique that adds a separate evaluator and web-search fallback. No fine-tuning needed — works with any LLM via prompting. Best for production systems where you want to handle retrieval failure gracefully without modifying the LLM itself. CRAG is much more practical for most teams.

Q2. What are the risks of agentic RAG and how do you handle them?
+

1. Infinite loops — the agent keeps retrieving without converging. Mitigation: set a max_iterations limit and a stopping criterion ("I have enough information to answer").

2. Tool misuse — agent calls the wrong tool or uses it incorrectly. Mitigation: clear tool descriptions, few-shot examples in the system prompt, input validation on tools.

3. Latency — each tool call adds round-trip time. Mitigation: parallelize tool calls when independent, use fast retrieval (ANN + cache), set timeouts.

4. Cost — many LLM calls per query. Mitigation: route simple queries to non-agentic path, use smaller models for planning steps.

Q3. What is the difference between Active RAG and Adaptive RAG?
+

Adaptive RAG focuses on the when — it decides upfront whether to retrieve, and if so, whether to do simple or complex retrieval. The routing decision happens once, before processing.

Active RAG focuses on dynamic, iterative retrieval — retrieval is triggered during generation whenever the model needs more information. It's more similar to IR-CoT: the model actively decides to retrieve mid-generation based on its current state of reasoning. Active RAG is more general; Adaptive RAG is specifically about routing the query complexity.

Retrieval Metrics — MRR and NDCG

Measuring ranking quality

These information-retrieval metrics measure how well the retrieval system ranks documents — independent of generation quality.

MRR — Mean Reciprocal Rank

For each query, find the rank of the first relevant doc.
If it's rank 1: score = 1.0
If it's rank 2: score = 0.5
If it's rank 5: score = 0.2
MRR = average of these reciprocal ranks across all queries.

NDCG — Normalized Discounted Cumulative Gain

Like MRR but handles graded relevance (relevant, very relevant, perfect). Documents ranked higher count more. Perfect ranking = NDCG of 1.0. Commonly used as NDCG@10 (top 10 results).

🧪

End-to-End vs Component Evaluation

Where is the pipeline failing?

A RAG pipeline can fail in different places. You need to evaluate each component to pinpoint the issue.

1

Retrieval quality: MRR, NDCG, Context Recall/Precision. "Are we retrieving the right docs?"

2

Generation quality: Faithfulness, Answer Relevance. "Is the LLM using the context correctly?"

3

End-to-end: RAGAS score, human eval, task-specific accuracy. "Does the full system answer correctly?"

💡

Interview tip: say "I'd first isolate whether it's a retrieval problem or a generation problem by evaluating them separately. Low context recall = fix retrieval. Low faithfulness = fix prompting or post-retrieval."

🔬

Faithfulness vs Hallucination

The most important safety metric

Faithfulness measures whether every claim in the generated answer is grounded in the retrieved context. A low faithfulness score means the LLM is making things up — hallucinating — even though it has context to work with.

Example

Context: "The company was founded in 2010 and has 500 employees."
Generated answer: "The company was founded in 2010, has 500 employees, and went public in 2018."

"Went public in 2018" is NOT in the context → faithfulness violation → hallucination.

RAGAS computes faithfulness by breaking the answer into claims, then verifying each claim against the context using an LLM judge.

Interview Q&A Evaluation Questions
Q1. How do you evaluate a RAG pipeline without human-labeled data?
+

Use LLM-as-judge frameworks like RAGAS. These use a powerful LLM (like GPT-4) to evaluate the pipeline's outputs automatically. For each query, RAGAS scores: faithfulness (are claims grounded?), answer relevance (does it answer the question?), context recall (did we retrieve the right docs?), and context precision (were retrieved docs relevant?).

You can also generate synthetic QA pairs from your documents using an LLM — these become your evaluation dataset. This is called synthetic data generation for RAG evaluation and is a common interview topic.

Q2. Your RAG system has high context recall but low faithfulness. What's the issue?
+

High context recall means retrieval is working — the right documents are being retrieved. Low faithfulness means the LLM is not using the retrieved context correctly — it's generating claims not supported by the context (hallucinating on top of good context).

Fixes: (1) Strengthen the system prompt: "Answer ONLY based on the provided context. Do not use prior knowledge." (2) Add a faithfulness check post-generation and re-generate if score is low. (3) Reduce the number of chunks — if there are 10 chunks, the LLM may ignore some and "fill in" from memory. (4) Use a model less prone to hallucination.

Q3. What is the difference between context recall and context precision?
+

Think of them like precision/recall in classification. Context recall: of all the information needed to answer the question (ground truth), how much was actually retrieved? Low recall means you missed important docs. Context precision: of all the documents you retrieved, how many were actually relevant? Low precision means you retrieved a lot of noise.

You want both high. Low recall → user gets incomplete answers. Low precision → LLM is distracted by irrelevant chunks, which hurts faithfulness.

Q4. How would you set up a RAG evaluation pipeline in production?
+

Step 1 — Create a golden dataset: 50–100 representative queries with expected answers and relevant document IDs (manually curated or LLM-generated).

Step 2 — Automate evaluation: Run RAGAS or a custom LLM judge on every deployment. Track faithfulness, context recall, answer relevance over time.

Step 3 — Monitor in production: Log queries, retrieved chunks, and generated answers. Sample 1–5% for human review. Track user feedback signals (thumbs up/down, follow-up queries indicating dissatisfaction).

Step 4 — Regression testing: Before deploying a new embedding model or chunking strategy, run the golden dataset and compare scores to the baseline. Never deploy a change that degrades RAGAS scores without understanding why.

Q5. Walk me through how you'd improve a RAG pipeline that gives wrong answers 30% of the time.
+

First, diagnose: Are wrong answers due to retrieval failure or generation failure? Run component-level eval. Check context recall — if the right docs aren't being retrieved, fix retrieval. Check faithfulness — if docs are retrieved but answer is wrong, fix generation.

If retrieval is the issue: Try hybrid search, HyDE, multi-query, better embeddings, or adjust chunk size. If generation is the issue: Improve system prompt, add contextual compression, add a faithfulness verifier, or try a stronger LLM.

Also check: Are the 30% wrong answers on a specific document type, query type, or topic? Pattern analysis often reveals a targeted fix (e.g., "all wrong answers involve tables → fix table chunking").
